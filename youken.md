# 障害対応訓練シミュレーション 要件定義書

## 障害対応訓練とは

実際に動いているサービス、または本番環境に近い検証環境に対して、意図的に障害を発生させ、それに対する調査・切り分け・復旧・報告を体験する訓練。

本サービスでは、実際に壊してよい sandbox 環境上に疑似サービスを構築し、プレイヤーが監視画面・ログ・ターミナル・Runbook などを使って障害対応を行う。

起こす障害は、アプリケーションの停止、ディスク枯渇、ネットワーク断、設定ミス、バッチ失敗などの真面目なものから、掃除のおばちゃんが電源を抜く、Red Bull で翼が生える、酒をキーボードにこぼすなどのふざけたものまで含める。

ただし、ふざけた障害であっても、対応プロセスとしては実際の障害対応に近い学びが得られるようにする。

## 想定している情景

主人公であるプレイヤーは、深夜の運用担当を任されている。

先輩エンジニアは寝ている。
上司は飲み会に行っている。
Runbook は古い。
Slack は燃えている。
監視画面は赤い。
朝までにサービスを復旧できなかったら解雇。

プレイヤーは、深夜 0 時から朝 6 時までの間に発生する様々な障害を乗り越え、一夜を生き延びることを目指す。

## テーマ

「うん」用

社内で謎に重要視されているサービス「うん用」を守る。

「うん用」は、社内業務、売上集計、通知、謎のバッチ処理などに使われている重要サービスという設定。

一部のシナリオでは、自作言語「うん言語」で書かれたバッチや設定ファイルが登場する。

## 想定するユーザー

主な対象ユーザーは以下。

* 障害対応訓練をしたことがない人
* 障害対応の進め方がわからない人
* SRE / インフラ / Web バックエンド運用に興味がある人
* ログやメトリクスの見方を学びたい人
* 「何から見ればいいのか」をゲーム形式で学びたい人
* コマンド操作や Linux の基本的な調査手順を実践したい人

対象ユーザーは必ずしもインフラ経験者ではないため、初級では丁寧なナビゲーションを入れる。

中級以降では少しずつナビゲーションを減らし、自力で仮説を立てる体験を重視する。

## 体験のゴール

プレイヤーに以下の障害対応の型を体験させる。

1. アラートに気づく
2. 影響範囲を確認する
3. メトリクスを見る
4. ログを見る
5. プロセス・ディスク・ネットワークなどを切り分ける
6. 暫定復旧する
7. 復旧確認をする
8. 原因を整理する
9. 必要に応じて報告する
10. 自分の対応を振り返る

単に「正解コマンドを打つゲーム」ではなく、「なぜそれを見るのか」「なぜその対応をするのか」を体験できるゲームにする。

## やりたいこと

Cloudflare Sandbox などを使って、実際にサービスが動いている環境を作成する。

その環境に対して、シナリオごとに様々な障害を発生させる。

プレイヤーは、FNaF のようなトリプルモニター風 UI を眺めながら、どんな障害が起こっているのか、どのように対応していくのかを判断する。

プレイヤーができることは以下。

* 監視モニターを見る
* ログを見る
* アラートを見る
* Runbook を読む
* ターミナルからコマンドを実行する
* 疑似 SSH でサーバーに入る
* プロセスを再起動する
* ファイルを確認・編集・削除する
* サービスの疎通確認をする
* Slack 風 UI で報告する
* 復旧完了を宣言する
* 終了後にリプレイを見る

## 重要機能: canvas 内録画・リプレイ

本サービスでは、プレイヤーの障害対応の様子を、ゲーム内 canvas の映像として録画する。

録画対象は、ゲーム UI として描画された canvas のみとする。

ブラウザ全体、別タブ、OS の通知、アドレスバー、開発者ツールなどは録画しない。

目的は以下。

* 自分の対応を後から見返せる
* どこで迷ったかを振り返れる
* 教材として使える
* 上級者の対応を他のユーザーが見られる
* マルチプレイ時の対応をチームで振り返れる
* SNS やコミュニティで共有できる
* 動画とタイムラインを見ながら対応の流れを理解できる

## 録画の基本方針

録画方式は canvas 内録画に絞る。

ゲーム画面の主要 UI を canvas に描画し、その canvas を録画する。

録画対象に含めるもの。

* 左モニターの監視画面
* 中央モニターのターミナル
* 右モニターの Runbook / Slack / チケット画面
* アラート演出
* カーソル位置
* クリック演出
* 入力中の文字
* 障害発生演出
* 復旧演出
* ゲーム内時刻
* シナリオ名
* 難易度
* REC 表示
* タイムライン用に重要なイベントの発生演出

録画対象に含めないもの。

* ブラウザのアドレスバー
* ユーザーの別タブ
* OS の通知
* 個人情報
* 開発者ツール
* 録画許可ダイアログ
* canvas 外の DOM UI

## canvas 内録画のメリット

canvas 内録画に絞る理由は以下。

* ゲーム画面だけを録画できる
* プライバシー面で安全
* 録画される範囲を完全に制御しやすい
* カーソルやクリック演出を自由に入れられる
* 録画映えする UI を作りやすい
* リプレイ動画として見やすい
* ブラウザ全体の画面共有に依存しない
* ユーザーが誤って別画面を録画する心配がない

## 録画開始・停止仕様

録画はシナリオ開始時に自動で開始する。

初回プレイ時には以下を明示する。

* ゲーム画面が録画されること
* 録画対象は canvas 内のゲーム画面のみであること
* ブラウザ全体や別タブは録画されないこと
* 録画は振り返りに使われること
* 公開するかどうかはユーザーが選べること
* 録画を保存しない設定も選べること

録画停止タイミング。

* シナリオクリア
* タイムアップ
* プレイヤーのリタイア
* 通信断
* ブラウザ終了
* 録画エラー

録画が失敗した場合でも、イベントログは可能な範囲で残す。

## 録画データ保存仕様

録画データは一定時間ごとに chunk として分割し、バックエンドにアップロードする。

例。

* 5 秒ごとに chunk 化
* 1 chunk ごとにアップロード
* セッション終了時に録画 metadata を保存
* 結果画面で動画として再生できるようにする

保存する metadata。

* replayId
* userId
* scenarioId
* difficulty
* startedAt
* finishedAt
* duration
* result
* videoObjectKey
* eventLogObjectKey
* thumbnailObjectKey
* visibility
* browserInfo
* recordingStatus

保存先の想定。

* 動画本体: R2 などのオブジェクトストレージ
* イベントログ: R2 または D1
* リプレイ metadata: D1
* プレイ中の一時状態: Durable Object

## イベントログ仕様

動画と同時に、タイムライン表示用のイベントログを timestamp 付きで保存する。

イベントログは採点のためではなく、リプレイ画面で動画と同期して「いつ何が起きたか」を表示するために使う。

保存するイベント例。

* session_start
* session_end
* scenario_event
* alert
* monitor_update
* terminal_input
* terminal_output
* command_detected
* ui_click
* ui_panel_open
* runbook_open
* slack_message_read
* file_opened
* service_restart
* recovery_check
* incident_resolved
* player_note

イベントログは JSONL 形式を基本とする。

例。

```jsonl id="jad278"
{"type":"session_start","at":0,"scenarioId":"disk-full-001"}
{"type":"alert","at":12000,"message":"HTTP 500 rate is above threshold"}
{"type":"ui_panel_open","at":15000,"panel":"metrics"}
{"type":"terminal_input","at":18000,"data":"df -h\n"}
{"type":"command_detected","at":18500,"command":"df -h"}
{"type":"terminal_output","at":19500,"data":"Filesystem Size Used Avail Use% Mounted on..."}
{"type":"ui_panel_open","at":26000,"panel":"runbook.disk-full"}
{"type":"terminal_input","at":40000,"data":"du -sh logs/*\n"}
{"type":"recovery_check","at":90000,"command":"curl localhost:3000/health","result":"ok"}
{"type":"incident_resolved","at":100000}
```

## イベントログでやりたいこと

イベントログを使って、リプレイ画面に以下を表示する。

* 障害が発生したタイミング
* アラートが鳴ったタイミング
* プレイヤーが開いたパネル
* 実行したコマンド
* Runbook を見たタイミング
* 復旧確認をしたタイミング
* 復旧宣言をしたタイミング

イベントログはあくまで振り返りの補助として使う。

プレイヤーに点数をつけるためには使わない。

## リプレイ画面仕様

リプレイ画面では、録画動画とタイムラインを同時に表示する。

表示内容。

* プレイ動画
* シナリオ名
* 難易度
* 成否
* 対応時間
* 発生アラート一覧
* 実行コマンド一覧
* 開いた Runbook
* 重要イベント一覧
* タイムライン

タイムライン例。

```txt id="ywielj"
00:00 シナリオ開始
00:12 HTTP 500 アラート発火
00:15 メトリクス画面を確認
00:18 df -h を実行
00:22 disk 100% を確認
00:26 disk full Runbook を開く
00:40 du -sh logs/* を実行
01:10 不要ログを削除
01:30 app restart
01:45 curl で復旧確認
02:00 復旧宣言
```

動画の該当時刻をクリックすると、タイムライン上のイベントも同期する。

タイムライン上のイベントをクリックすると、動画がその時刻にジャンプする。

リプレイには公開範囲を設定できる。

* 自分のみ
* URL を知っている人のみ
* チーム内公開
* 全体公開

## 想定障害

### 真面目な障害

* アプリケーションプロセス停止
* アプリケーション hang
* CPU 高騰
* メモリリーク
* no left space on device
* ログ肥大化
* DB 接続失敗
* DB connection pool 枯渇
* DNS 設定ミス
* ネットワーク断
* 証明書期限切れ
* 環境変数不足
* bad deploy
* rollback 失敗
* cron / batch 失敗
* キュー詰まり
* 外部 API 障害
* CDN キャッシュ不整合
* 監視自体の障害

### ふざけた障害

* サーバーの電源を掃除のおばちゃんが抜いちゃった
* no left space on device が起きて ssh できない
* ネットワークが切れたと思ったら LAN ケーブルで縄跳びしていた
* 障害対応するために Red Bull を飲んでいたら翼が生えて飛んでいく
* 冷蔵庫からお酒を取り出して飲んでいたらキーボードにこぼした
* 上司が「再起動すれば直る」とだけ Slack に書き残して寝る
* Runbook が全部「気合いで直す」と書かれている
* うん言語のエラーが「うんともすんとも」しか出ない
* 掃除のおばちゃんが冗長系の両方の電源を抜く
* アラート音がうるさすぎて逆に重要なアラートを見落とす

## ゲームフロー

### 1. 難易度選択

初級、中級、上級の 3 つを用意する。

### 2. シナリオ選択

難易度ごとに複数のシナリオを用意する。

初回プレイではチュートリアルシナリオを推奨する。

### 3. ブリーフィング

開始前に、以下を表示する。

* サービス概要
* 現在の構成
* 重要な URL
* 使えるコマンド
* 参考 Runbook
* 今夜の運用目標
* 録画されることの説明
* 録画対象が canvas 内だけであることの説明

### 4. プレイ開始

ゲーム内時刻が進み、障害が発生する。

プレイヤーは監視モニター・ターミナル・Runbook などを使って障害対応する。

シナリオ開始と同時に canvas 内録画を開始する。

### 5. 障害発生

アラート、ログ、メトリクス、Slack 風通知などで異常を知らせる。

障害はシナリオ定義に従って発生する。

### 6. 調査・切り分け

プレイヤーはコマンドや UI 操作で原因を探る。

### 7. 暫定復旧

プレイヤーはプロセス再起動、設定修正、ログ削除、rollback などで復旧を試みる。

### 8. 復旧確認

プレイヤーは curl、監視画面、ログなどで復旧を確認する。

### 9. 完了宣言

プレイヤーが「復旧完了」を宣言する。

システムが成功条件を満たしているか確認する。

### 10. 結果画面

以下を表示する。

* 成功 / 失敗
* 対応時間
* 発生した障害
* 実行コマンド一覧
* 開いた Runbook
* 重要イベント一覧
* リプレイ動画
* タイムライン
* 再挑戦ボタン

## 難易度

### 初級

チュートリアル的にナビゲーションを入れる。

特徴。

* 障害原因は単純
* ログにヒントが多い
* Runbook が正しい
* 監視画面がわかりやすい
* 危険操作をしそうなときに警告が出る
* 何を見るべきかを段階的に案内する

障害例。

* プロセスが止まった
* アプリが hang した
* ディスクがいっぱいになった
* DB が起動していない
* 環境変数が足りない

### 中級

少しずつ自力で判断させる。

特徴。

* 原因が一発ではわからない
* 複数のメトリクスを見る必要がある
* Runbook がやや古い
* 一時復旧しても再発することがある
* ノイズアラートが混ざる

障害例。

* DB connection pool 枯渇
* bad deploy
* DNS ミス
* ログ肥大化による disk full
* メモリリーク
* CDN キャッシュ不整合

### 上級

理不尽だが、完全な運ゲーにはしない。

特徴。

* 複合障害
* 監視自体が壊れる
* Runbook が古い
* ノイズが多い
* 時間制限が厳しい
* 安易な操作で被害が広がる
* コミュニケーションも重要になる

障害例。

* disk full + 再起動ループ
* DNS キャッシュ問題
* rollback したら別の障害
* 監視メトリクスが欠損
* 掃除のおばちゃんが両系の電源を抜く
* うん言語バッチが謎のエラーを出す

## シナリオ定義

シナリオは YAML または JSON で定義する。

例。

```yaml id="zp5xtd"
id: disk-full-001
title: "ログが世界を埋め尽くす夜"
difficulty: beginner
time_limit_minutes: 12

service:
  name: "うん用 API"
  url: "http://localhost:3000"

initial_state:
  app: running
  db: running
  disk_usage: 70

triggers:
  - at: 60
    type: append_file
    target: "/var/log/unyoh/debug.log"
    size_mb: 800

alerts:
  - at: 90
    type: monitor
    message: "HTTP 500 rate is above threshold"
  - at: 120
    type: monitor
    message: "Disk usage is above 95%"

success_conditions:
  - type: http_status
    url: "http://localhost:3000/health"
    status: 200
  - type: disk_usage_below
    value: 80
  - type: app_running
    service: "unyoh-api"

timeline_events:
  - id: disk-alert
    at: 120
    label: "Disk usage alert"
  - id: app-recovered
    condition: "success_conditions_met"
    label: "Service recovered"
```

## うん言語

### うん言語とは

「うん」用の一部バッチや設定に使われている謎の自作言語。

ふざけた見た目だが、障害対応では実際に仕様を読んで原因を特定する必要がある。

### 仕様案

| 構文        | 意味                  |
| --------- | ------------------- |
| うんちく      | コメント                |
| うん <name> | 変数・定義               |
| うーん       | 計算                  |
| うん？       | 条件                  |
| うん！       | return              |
| うんわり      | 割り算                 |
| うんたす      | 足し算                 |
| うんかけ      | 掛け算                 |
| うんなし      | 0 / false           |
| うんあり      | 1 / true            |
| うんともすんとも  | error / unreachable |

### うん言語シナリオ例

```txt id="i3rsx0"
うんちく 売上集計バッチ
うん x = 100
うん y = うんなし
うん z = x うんわり y
うん！
```

発生する障害。

* 深夜 3 時の集計バッチが失敗
* ログには「うんともすんとも」しか出ない
* プレイヤーは仕様表を読んで 0 除算に気づく
* バッチを止める、修正する、または暫定対応する

### Investigator

* ログを見る

* 仮説を立てる

* 復旧宣言の瞬間がわかる

* シナリオ名

* 現在の状態
* アラート数

### 録画

MVP から canvas 内録画機能を入れる。

最低限の録画仕様。

* シナリオ開始時に録画開始
* シナリオ終了時に録画停止
* canvas 内だけを録画
* 録画データを保存
* 結果画面で再生
* イベントログとタイムラインを併用
* 録画失敗時もイベントログは可能な範囲で残す

### リプレイ

最低限のリプレイ。

* 動画再生
* タイムライン表示
* 動画とタイムラインの同期
* 実行コマンド一覧
* アラート一覧
* 重要イベント一覧

## 非機能要件

### セキュリティ

* Sandbox はユーザーごとに分離する
* 他ユーザーの環境にアクセスできないようにする
* コマンド実行には制限を設ける
* 外部ネットワークアクセスは必要最小限にする
* 危険な操作はシナリオ内で制御する
* 動画公開範囲を制御する
* 録画前にユーザーへ説明する
* 録画対象を canvas 内に限定する
* 個人情報を録画しない設計にする

### パフォーマンス

* 録画中でもゲーム操作が重くならないこと
* canvas 描画と録画処理が重くなりすぎないこと
* chunk upload により終了時の待ち時間を減らすこと
* メトリクス更新で UI が固まらないこと
* ターミナル出力が遅延しすぎないこと

### 可用性

* 録画が失敗してもプレイは継続できること
* Sandbox が落ちた場合はエラーを表示すること
* 通信断時には可能な範囲でイベントログを保持すること
* シナリオ途中離脱時にも partial replay を残せること

### 拡張性

* シナリオを追加しやすいこと
* 障害注入をプラグイン化できること
* タイムラインイベントをシナリオごとに変更できること
* うん言語シナリオを追加できること
* マルチプレイに拡張できること
* リプレイ動画にコメントを追加できること

* 動画とタイムラインで自分の対応を振り返れる

## 一言で言うと

深夜の監視室で、謎の社内サービス「うん用」を守る、FNaF 風の障害対応訓練ゲーム。

プレイヤーの対応は canvas 内のゲーム画面として録画され、終了後に動画とタイムラインを並べて振り返ることができる。

採点はせず、プレイヤー自身がリプレイを見ながら「どこで迷ったか」「何を見落としたか」「どう復旧したか」を振り返る体験を重視する。
