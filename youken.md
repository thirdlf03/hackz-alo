# 障害対応訓練シミュレーション 要件定義書

このドキュメントはプロダクトの正(SSoT)である。世界観・仕様に関してはコードよりもこのドキュメントを優先する。コード側の識別子はまだ旧名称のまま残っている箇所があるが、移行対象は「旧名称からの移行メモ」節にまとめる。

## ポジショニング

GameDay(障害対応訓練)をやりたいが、準備コストが高くて手が出せないチームのための、準備ゼロで実施できる訓練環境。

本サービスは 2 つのモードを持つ。ソロモードは、個人が手軽に障害対応の「型」を体験するための入口。マルチプレイ(Exercise Room)は、ロールを分担して行う blameless な本格チーム訓練の場。どちらも、壊してよい sandbox 上に疑似サービスを立て、監視・ログ・ターミナル・Runbook を使って調査から復旧・報告までを一貫して体験できる。訓練の準備(環境構築・シナリオ作成・進行管理)は本サービスが肩代わりする。

## 障害対応訓練とは

実際に動いているサービス、または本番環境に近い検証環境に対して、意図的に障害を発生させ、それに対する調査・切り分け・復旧・報告を体験する訓練。

本サービスでは、実際に壊してよい sandbox 環境上に疑似サービスを構築し、プレイヤーが監視画面・ログ・ターミナル・Runbook などを使って障害対応を行う。

起こす障害は、アプリケーションの停止、ディスク枯渇、ネットワーク断、設定ミス、バッチ失敗などの真面目なものから、掃除のおばちゃんが電源を抜く、LAN ケーブルで縄跳びしていてネットワークが切れる、キーボードに飲み物をこぼす、といった現場あるある寄りのものまで含める。

ただし、あるある寄りの障害であっても、対応プロセスとしては実際の障害対応に近い学びが得られるようにする。

## 世界観・トーンガイド

### ユーモアは「名前」ではなく「状況」に宿す

キャラクターや固有名詞で笑わせにいくのではなく、現場で誰もが「あるある」と頷く状況にユーモアを宿す。障害対応そのものは真面目に扱い、その周辺の人間ドラマを軽く描く。

残す・推奨する現場あるある。

- Runbook が古い、または「気合いで直す」しか書かれていない
- 先輩エンジニアは寝ている
- 上司は飲み会に行っていて「再起動すれば直る」とだけ書き残す
- 掃除のおばちゃんが電源を抜く
- LAN ケーブルで縄跳びしていてネットワークが切れる
- エナジードリンクで夜通し戦う
- キーボードに飲み物をこぼす
- アラート音がうるさすぎて逆に重要なアラートを見落とす

### トーン

目指すトーンは「本物の緊張感 + くすっとくる現場あるある」。監視室に一人、モニターが赤い、朝までに直さないといけない、という本物の緊張感を土台にしつつ、状況のあるあるで肩の力を抜けるようにする。緊張感は「朝 6 時までの復旧」と「サービス影響時間(ダウンタイム)の記録」で担保する。

### 禁止事項

- 下ネタ・内輪ネタ
- 実在の人物・企業・製品への言及
- 個人を責める演出(解雇・懲罰・叱責など)
- チーム訓練の場で気まずくなる表現

## 想定している情景

主人公であるプレイヤーは、深夜の運用担当を任されている。

暗い深夜の NOC(監視室)で、トリプルモニターを見つめる。

先輩エンジニアは寝ている。
上司は飲み会に行っている。
Runbook は古い。
チャットは燃えている。
監視画面は赤い。

プレイヤーは、深夜 0 時から朝 6 時までの間に発生する様々な障害を乗り越え、朝までにサービスを復旧して一夜を守り抜くことを目指す。

## テーマ

社内基幹サービス「やまびこ」(英字識別子は `yamabiko`)を守る。

「やまびこ」は、社内業務、売上集計、通知、そして誰も全容を把握していない謎のバッチ処理に使われている、レガシーな基幹サービスという設定。和名のコードネームが付いているのは、日本企業あるあるとして意図的なもの。動いてはいるが、なぜ動いているのかを説明できる人はもういない。

一部のシナリオでは、社内 DSL「こだま」(英字識別子は `kodama`)で書かれたバッチや設定ファイルが登場する。

## エンディング仕様

本サービスに「解雇」エンドは存在しない。成否は 2 種類の「朝の情景」で表現する。どちらのエンドでも、リプレイと振り返りを通じて学びに変換できる blameless な設計とする。

### 静かな朝(成功)

朝までに復旧条件を満たした場合。何事もなかったかのように日勤帯へ引き継いで退勤する。静かな朝が明ける。

### にぎやかな朝(失敗)

朝までに復旧条件を満たせなかった場合。朝会で共有されるインシデントレポートが発行される。失敗は責めるためではなく、次の訓練に活かすために記録する。

インシデントレポートに含める内容。

- サービス影響時間(ダウンタイム)
- 対応タイムライン
- 良かった点
- 次に試すこと

失敗しても、リプレイと振り返りで「どこで迷ったか」「何を見落としたか」を確認し、もう一度挑戦できる。

## 想定するユーザー

主な対象ユーザーは以下。

- 障害対応訓練をしたことがない人
- 障害対応の進め方がわからない人
- SRE / インフラ / Web バックエンド運用に興味がある人
- ログやメトリクスの見方を学びたい人
- 「何から見ればいいのか」をゲーム形式で学びたい人
- コマンド操作や Linux の基本的な調査手順を実践したい人

また、チームでの利用者も主要な対象とする。

- SRE チーム・開発チームの訓練担当
- EM(エンジニアリングマネージャー)
- 新人研修の担当者

対象ユーザーは必ずしもインフラ経験者ではないため、初級では丁寧なナビゲーションを入れる。

中級以降では少しずつナビゲーションを減らし、自力で仮説を立てる体験を重視する。

## 体験のゴール

プレイヤーに以下の障害対応の型を体験させる。

1. アラートに気づく
2. 影響範囲を確認する
3. メトリクスを見る
4. ログを見る
5. プロセス・ディスク・ネットワークなどを切り分ける
6. 暫定復旧する
7. 復旧確認をする
8. 原因を整理する
9. 必要に応じて報告する
10. 自分の対応を振り返る

単に「正解コマンドを打つゲーム」ではなく、「なぜそれを見るのか」「なぜその対応をするのか」を体験できるゲームにする。

マルチプレイ(Exercise Room)では、上記に加えて以下のチーム対応の型も体験させる。

- 役割分担する
- 宣言してから操作する
- 状況共有の型を身につける

## やりたいこと

Cloudflare Sandbox などを使って、実際にサービスが動いている環境を作成する。

その環境に対して、シナリオごとに様々な障害を発生させる。

プレイヤーは、暗い深夜の NOC(監視室)を模したトリプルモニター風 UI を眺めながら、どんな障害が起こっているのか、どのように対応していくのかを判断する。

プレイヤーができることは以下。

- 監視モニターを見る
- ログを見る
- アラートを見る
- Runbook を読む
- ターミナルからコマンドを実行する
- 疑似 SSH でサーバーに入る
- プロセスを再起動する
- ファイルを確認・編集・削除する
- サービスの疎通確認をする
- チャット風 UI で報告する
- 復旧完了を宣言する
- 終了後にリプレイを見る

## 重要機能: canvas 内録画・リプレイ

本サービスでは、プレイヤーの障害対応の様子を、ゲーム内 canvas の映像として録画する。

録画対象は、ゲーム UI として描画された canvas のみとする。

ブラウザ全体、別タブ、OS の通知、アドレスバー、開発者ツールなどは録画しない。

目的は以下。

- 自分の対応を後から見返せる
- どこで迷ったかを振り返れる
- 教材として使える
- 上級者の対応を他のユーザーが見られる
- マルチプレイ時の対応をチームで振り返れる
- コミュニティで共有できる
- 動画とタイムラインを見ながら対応の流れを理解できる

## 録画の基本方針

録画方式は canvas 内録画に絞る。

ゲーム画面の主要 UI を canvas に描画し、その canvas を録画する。

録画対象に含めるもの。

- 左モニターの監視画面
- 中央モニターのターミナル
- 右モニターの Runbook / チャット / チケット画面
- アラート演出
- カーソル位置
- クリック演出
- 入力中の文字
- 障害発生演出
- 復旧演出
- ゲーム内時刻
- シナリオ名
- 難易度
- REC 表示
- タイムライン用に重要なイベントの発生演出

録画対象に含めないもの。

- ブラウザのアドレスバー
- ユーザーの別タブ
- OS の通知
- 個人情報
- 開発者ツール
- 録画許可ダイアログ
- canvas 外の DOM UI

## canvas 内録画のメリット

canvas 内録画に絞る理由は以下。

- ゲーム画面だけを録画できる
- プライバシー面で安全
- 録画される範囲を完全に制御しやすい
- カーソルやクリック演出を自由に入れられる
- 録画映えする UI を作りやすい
- リプレイ動画として見やすい
- ブラウザ全体の画面共有に依存しない
- ユーザーが誤って別画面を録画する心配がない

## 録画開始・停止仕様

録画はシナリオ開始時に自動で開始する。

初回プレイ時には以下を明示する。

- ゲーム画面が録画されること
- 録画対象は canvas 内のゲーム画面のみであること
- ブラウザ全体や別タブは録画されないこと
- 録画は振り返りに使われること
- 録画を保存しない設定も選べること

録画停止タイミング。

- シナリオクリア
- タイムアップ
- プレイヤーのリタイア
- 通信断
- ブラウザ終了
- 録画エラー

録画が失敗した場合でも、イベントログは可能な範囲で残す。

## 録画データ保存仕様

録画データは一定時間ごとに chunk として分割し、バックエンドにアップロードする。

例。

- 5 秒ごとに chunk 化
- 1 chunk ごとにアップロード
- セッション終了時に録画 metadata を保存
- 結果画面で動画として再生できるようにする

保存する metadata。

- replayId
- scenarioId
- difficulty
- startedAt
- finishedAt
- duration
- result
- videoObjectKey
- eventLogObjectKey
- thumbnailObjectKey
- browserInfo
- recordingStatus

保存先の想定。

- 動画本体: R2 などのオブジェクトストレージ
- イベントログ: R2 または D1
- リプレイ metadata: D1
- プレイ中の一時状態: Durable Object

## イベントログ仕様

動画と同時に、タイムライン表示用のイベントログを timestamp 付きで保存する。

イベントログは採点のためではなく、リプレイ画面で動画と同期して「いつ何が起きたか」を表示するために使う。

保存するイベント例。

- session_start
- session_end
- scenario_event
- alert
- monitor_update
- terminal_input
- terminal_output
- command_detected
- ui_click
- ui_panel_open
- runbook_open
- chat_message_read
- file_opened
- service_restart
- recovery_check
- incident_resolved
- player_note

イベントログは JSONL 形式を基本とする。

例。

```jsonl id="jad278"
{"type":"session_start","at":0,"scenarioId":"disk-full-001"}
{"type":"alert","at":12000,"message":"HTTP 500 rate is above threshold"}
{"type":"ui_panel_open","at":15000,"panel":"metrics"}
{"type":"terminal_input","at":18000,"data":"df -h\n"}
{"type":"command_detected","at":18500,"command":"df -h"}
{"type":"terminal_output","at":19500,"data":"Filesystem Size Used Avail Use% Mounted on..."}
{"type":"ui_panel_open","at":26000,"panel":"runbook.disk-full"}
{"type":"terminal_input","at":40000,"data":"du -sh logs/*\n"}
{"type":"recovery_check","at":90000,"command":"curl localhost:3000/health","result":"ok"}
{"type":"incident_resolved","at":100000}
```

## イベントログでやりたいこと

イベントログを使って、リプレイ画面に以下を表示する。

- 障害が発生したタイミング
- アラートが鳴ったタイミング
- プレイヤーが開いたパネル
- 実行したコマンド
- Runbook を見たタイミング
- 復旧確認をしたタイミング
- 復旧宣言をしたタイミング

イベントログはあくまで振り返りの補助として使う。

プレイヤーに点数をつけるためには使わない。

## リプレイ画面仕様

リプレイ画面では、録画動画とタイムラインを同時に表示する。

表示内容。

- プレイ動画
- シナリオ名
- 難易度
- 成否
- 対応時間
- 発生アラート一覧
- 実行コマンド一覧
- 開いた Runbook
- 重要イベント一覧
- タイムライン

タイムライン例。

```txt id="ywielj"
00:00 シナリオ開始
00:12 HTTP 500 アラート発火
00:15 メトリクス画面を確認
00:18 df -h を実行
00:22 disk 100% を確認
00:26 disk full Runbook を開く
00:40 du -sh logs/* を実行
01:10 不要ログを削除
01:30 app restart
01:45 curl で復旧確認
02:00 復旧宣言
```

動画の該当時刻をクリックすると、タイムライン上のイベントも同期する。

タイムライン上のイベントをクリックすると、動画がその時刻にジャンプする。

リプレイにはコメントを付けられる。チームでの振り返りや、名リプレイの共有に使う。

## 想定障害

### 真面目な障害

- アプリケーションプロセス停止
- アプリケーション hang
- CPU 高騰
- メモリリーク
- no left space on device
- ログ肥大化
- DB 接続失敗
- DB connection pool 枯渇
- DNS 設定ミス
- ネットワーク断
- 証明書期限切れ
- 環境変数不足
- bad deploy
- rollback 失敗
- cron / batch 失敗
- キュー詰まり
- 外部 API 障害
- CDN キャッシュ不整合
- 監視自体の障害

### 現場あるある寄りの障害

- サーバーの電源を掃除のおばちゃんが抜いてしまった
- no left space on device が起きて ssh できない
- ネットワークが切れたと思ったら、誰かが LAN ケーブルで縄跳びしていた
- エナジードリンクを飲みながら夜通し戦っていたら、キーボードに飲み物をこぼした
- 上司が「再起動すれば直る」とだけチャットに書き残して寝る
- Runbook が全部「気合いで直す」と書かれている
- 「こだま」バッチのエラーが「こだまが返ってきません」しか出ない
- 掃除のおばちゃんが冗長系の両方の電源を抜く
- アラート音がうるさすぎて逆に重要なアラートを見落とす

## ゲームフロー

実装済みの画面フローは以下。

### 1. 難易度選択

初級、中級、上級の 3 つから選ぶ。

### 2. シナリオ一覧

難易度ごとに複数のシナリオを表示する。

初回プレイではチュートリアルシナリオを推奨する。

### 3. ロビー

シナリオ選択後、ソロ・マルチを問わず必ずロビーを通る。ここで表示名を確認し、ロールを決める。ソロでも表示名・ロール確認の場として表示される。マルチプレイ(Exercise Room)では、参加者どうしでロールを分担する。

### 4. ブリーフィング(録画同意)

ロビーから進むと、開始前に以下を表示する。

- サービス概要
- 現在の構成
- 重要な URL
- 使えるコマンド
- 参考 Runbook
- 今夜の運用目標
- 録画されることの説明
- 録画対象が canvas 内だけであることの説明

ここでプレイヤーは録画について同意する。

### ページャー通知

ブリーフィング画面には「📟 ページャー待機」ボタンがあり、任意でブラウザ通知(Web Push)を許可できる。

許可すると、訓練開始の合図と同時にスマホや PC に「【P1】」から始まる通知が届く。

タイトルはシナリオ名、本文は今夜の障害の概要と「オンコールはあなた一人、朝 6 時までに復旧せよ」という一文。

画面を閉じていても、別タブを見ていても、その瞬間から障害対応が始まったことを突きつけてくる、本物のページャーさながらの演出。

通知をタップすると訓練画面に戻れる。

許可しなくても訓練の進行・成否には一切影響しない。あくまで臨場感を高めるための任意機能。

### 5. プレイ

暗い深夜の NOC を模したトリプルモニター canvas を中心に、以下のパネルで対応する。

- Team パネル(参加者とロール)
- Tasks パネル
- Incident Log パネル
- Injects パネル(進行役が投入するインジェクト)

ゲーム内時刻が進み、障害が発生する。プレイヤーは監視モニター・ターミナル・Runbook などを使って障害対応する。シナリオ開始と同時に canvas 内録画を開始する。

### 6. Hotwash(ふりかえり)

プレイ終了後、3 問のふりかえりに答え、AAR(After Action Report)を生成する。

### 7. 結果

成否や対応時間、実行コマンド、開いた Runbook、重要イベントなどを表示する。

### 8. リプレイ

録画動画とタイムラインを同期して表示する。コメントを付けて振り返れる。

## 難易度

### 初級

チュートリアル的にナビゲーションを入れる。

特徴。

- 障害原因は単純
- ログにヒントが多い
- Runbook が正しい
- 監視画面がわかりやすい
- 危険操作をしそうなときに警告が出る
- 何を見るべきかを段階的に案内する

障害例。

- プロセスが止まった
- アプリが hang した
- ディスクがいっぱいになった
- DB が起動していない
- 環境変数が足りない

### 中級

少しずつ自力で判断させる。

特徴。

- 原因が一発ではわからない
- 複数のメトリクスを見る必要がある
- Runbook がやや古い
- 一時復旧しても再発することがある
- ノイズアラートが混ざる

障害例。

- DB connection pool 枯渇
- bad deploy
- DNS ミス
- ログ肥大化による disk full
- メモリリーク
- CDN キャッシュ不整合

### 上級

理不尽だが、完全な運ゲーにはしない。

特徴。

- 複合障害
- 監視自体が壊れる
- Runbook が古い
- ノイズが多い
- 時間制限が厳しい
- 安易な操作で被害が広がる
- コミュニケーションも重要になる

障害例。

- disk full + 再起動ループ
- DNS キャッシュ問題
- rollback したら別の障害
- 監視メトリクスが欠損
- 掃除のおばちゃんが両系の電源を抜く
- 「こだま」バッチが謎のエラーを出す

## シナリオ定義

シナリオは YAML または JSON で定義する。

例。

```yaml id="zp5xtd"
id: disk-full-001
title: 'ログが世界を埋め尽くす夜'
difficulty: beginner
time_limit_minutes: 12

service:
  name: 'やまびこ API'
  url: 'http://localhost:3000'

initial_state:
  app: running
  db: running
  disk_usage: 70

triggers:
  - at: 60
    type: append_file
    target: '/var/log/yamabiko/debug.log'
    size_mb: 800

alerts:
  - at: 90
    type: monitor
    message: 'HTTP 500 rate is above threshold'
  - at: 120
    type: monitor
    message: 'Disk usage is above 95%'

success_conditions:
  - type: http_status
    url: 'http://localhost:3000/health'
    status: 200
  - type: disk_usage_below
    value: 80
  - type: app_running
    service: 'yamabiko-api'

timeline_events:
  - id: disk-alert
    at: 120
    label: 'Disk usage alert'
  - id: app-recovered
    condition: 'success_conditions_met'
    label: 'Service recovered'
```

## こだま(社内 DSL)

### こだまとは

「やまびこ」の一部のバッチや設定に使われている、社内 DSL。退職した先代エンジニアが残した自作バッチ言語で、仕様書は本人の個人 wiki ごと消えている。ゲーム内の仕様表だけが頼り。

見た目はとぼけているが、障害対応では実際に仕様を読んで原因を特定する必要がある。エラーが起きても「こだまが返ってきません」しか出ないため、仕様表を読み解いて自力で原因にたどり着く。

### 仕様案

| 構文                   | 意味                |
| ---------------------- | ------------------- |
| やまびこ帳             | コメント            |
| よぶ <name>            | 変数・定義          |
| はかる                 | 計算                |
| もし？                 | 条件                |
| かえす                 | return              |
| わる                   | 割り算              |
| たす                   | 足し算              |
| かける                 | 掛け算              |
| しずか                 | 0 / false           |
| こだま                 | 1 / true            |
| こだまが返ってきません | error / unreachable |

### こだまシナリオ例

```txt id="i3rsx0"
やまびこ帳 売上集計バッチ
よぶ x = 100
よぶ y = しずか
よぶ z = x わる y
かえす
```

発生する障害。

- 深夜 3 時の集計バッチが失敗
- ログには「こだまが返ってきません」しか出ない
- プレイヤーは仕様表を読んで 0 除算に気づく
- バッチを止める、修正する、または暫定対応する

## マルチプレイ(Exercise Room)

マルチプレイは Exercise Room として実装済み。ロールを分担して、blameless な本格チーム訓練を行う。

- コメント

## デザイン

暗い深夜の NOC(監視室)をイメージする。プレイヤーは基本的にトリプルモニターを見る。

トーンの参考として、監視カメラ的な不穏さ(深夜、赤いモニター、静かな緊張)と、朝が来たときの安堵の対比を演出に取り入れる。

### 左モニター

監視・メトリクス。

表示するもの。

- CPU
- Memory
- Disk
- HTTP 5xx
- Latency
- RPS
- DB connections
- Queue depth
- Alert list
- サービス依存トポロジー(生死マップ)

### サービス依存マップ(生死マップ)

シナリオは `topology`(ノードとエッジからなる依存グラフ)を定義できる。ノードは `user` を起点とし、実プロセスに紐付くノードは `startup` の id を参照する。エッジは「呼び出し元 → 呼び出し先」の向きで、呼び出し元が呼び出し先に依存することを表す。

障害注入によって特定のプロセスが停止すると、対応するノードが down になり、そのノードに依存する上流ノード(エッジの `from` 側)は degraded として表示される。復旧判定(success conditions)が満たされると、該当ノードは healthy に戻る。

このマップは左モニターに常時表示され、監視メトリクスと並んで「どこが壊れていて、どこに影響が波及しているか」を一目で把握できるようにする。

### 中央モニター

自由操作画面。

表示するもの。

- ターミナル
- 疑似 SSH
- ログ tail
- エディタ
- サービス状態
- 復旧ボタン

### 右モニター

情報・ノイズ・演出。

表示するもの。

- Runbook
- チャット風通知
- チケット
- 上司からのメッセージ
- 掃除のおばちゃんカメラ
- 休憩室カメラ
- エナジードリンク残量
- 「こだま」仕様表

## 非機能要件

### セキュリティ

- Sandbox はプレイセッションごとに分離する
- コマンド実行には制限を設ける
- 外部ネットワークアクセスは必要最小限にする
- 危険な操作はシナリオ内で制御する
- 録画前にユーザーへ説明する
- 録画対象を canvas 内に限定する
- 個人情報を録画しない設計にする

### パフォーマンス

- 録画中でもゲーム操作が重くならないこと
- canvas 描画と録画処理が重くなりすぎないこと
- chunk upload により終了時の待ち時間を減らすこと
- メトリクス更新で UI が固まらないこと
- ターミナル出力が遅延しすぎないこと

### 可用性

- 録画が失敗してもプレイは継続できること
- Sandbox が落ちた場合はエラーを表示すること
- 通信断時には可能な範囲でイベントログを保持すること
- シナリオ途中離脱時にも partial replay を残せること

### 拡張性

- シナリオを追加しやすいこと
- 障害注入をプラグイン化できること
- タイムラインイベントをシナリオごとに変更できること
- 「こだま」シナリオを追加できること
- マルチプレイに拡張できること
- リプレイ動画にコメントを追加できること

## 一言で言うと

深夜の監視室で、社内の誰も全容を知らない基幹サービス「やまびこ」を朝まで守り抜く、障害対応訓練シミュレーター。

プレイヤーの対応は canvas 内のゲーム画面として録画され、終了後に動画とタイムラインを並べて振り返ることができる。ソロモードでは自分の対応を、マルチプレイ(Exercise Room)では blameless な本格チーム訓練を通じて、チームで振り返れる。

採点はせず、プレイヤー自身がリプレイを見ながら「どこで迷ったか」「何を見落としたか」「どう復旧したか」を振り返る体験を重視する。
